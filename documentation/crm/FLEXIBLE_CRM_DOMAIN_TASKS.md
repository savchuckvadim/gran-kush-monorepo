# Гибкая multi-tenant CRM: доменные задачи (roadmap)

Документ фиксирует **расширенную** постановку по полям, статусам, настройкам портала, товарам/остаткам, финансам, посещаемости и интеграции с внешними сайтами клубов. Задачи разбиты на **бэкенд** и **фронт (CRM + при необходимости LK)** с охватом слоёв: **DB → Prisma → domain → services → DTO → controllers → UI**.

Главная цель: сделать CRM, которая:

- работает из коробки для нового клуба (seed + разумные дефолты);
- гибко кастомизируется под любой клуб без форков кода;
- сохраняет целостность данных и аналитики даже при скрытии/архивации сущностей;
- не ломает tenant-изоляцию на API и UI.

**Связанные документы (уже в репозитории):**

- [MULTI_TENANT_MICROSAAS_PLAN.md](./MULTI_TENANT_MICROSAAS_PLAN.md) — §8 Dynamic Fields (базовая идея metadata-driven полей)
- [AUTHENTICATION.md](../AUTHENTICATION.md), [HTTP_API_CONTRACT.md](../backend/HTTP_API_CONTRACT.md) — портал, куки, мобильный Bearer
- [CRM_IMPLEMENTATION_PLAN.md](./CRM_IMPLEMENTATION_PLAN.md) — attendance и прочее

## 1. Срез «что уже есть» (apps/api + Prisma)

| Область | Сейчас в схеме / коде | Замечание |
|--------|------------------------|-----------|
| Портал | `Portal`, `PortalTypeEnum`, статус портала | База для tenant |
| Member | Фиксированные колонки (`name`, `phone`, `status` string и т.д.), `portalId` | Нет универсальных custom fields |
| Product | `Product`, `ProductCategory`, `MeasurementUnit`, `initialQuantity` / `currentQuantity` | **Нет `portalId` у товара** — каталог сейчас фактически глобальный; для клубной CRM нужна изоляция по порталу |
| Order / OrderItem | Есть, связь с `member`, `product` | Статусы заказа — строки в модели |
| Finance | `FinancialTransaction` | Суммы, тип, связь с order/member; **нет строкового снапшота товара** в транзакции — при удалении продукта история может терять читаемость |
| Presence | `PresenceSession`, QR | Основа для аналитики посещений |
| Документы member | `Document`, `MemberDocument`, `IdentityDocument` | Глобальный справочник `Document`; требования «какие документы обязательны» — в настройках портала ещё не выражены |
| Auth / tenant | `PortalContextMiddleware`, `X-Portal-Id` / `X-Portal-Slug`, `PortalTenantMatchGuard` | Контекст портала на `/lk/*`, `/crm/*`, `/users/*` |

**CRM (`apps/crm`):** роутинг с `[portal]`, cookie-auth к API, формы регистрации member/employee с заголовком `x-portal-slug`, списки members/products/orders/attendance — без универсального движка полей/статусов.

### Почему этого уже недостаточно

- Для 2-3 клубов на фиксированной схеме это ок, но при масштабировании каждый клуб попросит свои поля/статусы/скрытые параметры.
- Свободные строковые `status` и ручные правки количества товара быстро приводят к хаосу в отчетах.
- Отсутствие части tenant-scoping в каталоге (`Product.portalId`) создаёт риск утечек между порталами.
- Нет единой модели конфигурации UI (что показывать в карточках/таблицах/фильтрах).

## 2. Epic A — Внешний сайт клуба: регистрация, вход, ЛК (интеграция)

**Цель:** клуб встраивает ссылку или виджет (Tilda / Vue / любой стек); пользователь проходит регистрацию/логин и попадает в ЛК на **нашем домене**.

**Технические принципы (зафиксировать в реализации):**

- **HttpOnly + SameSite=Lax** (см. `ConfigCookieService`) — полноценная сессия в **iframe на чужом домене** ненадёжна (third-party cookies). Рекомендуемый UX: **редирект на top-level** (`window.location`) на `https://<member-app>/{portal}/login|register`), опционально `returnUrl` с whitelist / подписью.
- Альтернатива для встраивания без редиректа: **только регистрация** в iframe + `postMessage` родителю; логин и ЛК — редирект.
- Публичные ссылки для клуба: шаблоны URL с `portalSlug` (или signed token), раздел в CRM «Интеграция»: скопировать ссылку / HTML snippet.

**Задачи (кратко):**

| Слой | Backend | Frontend (CRM + LK) |
|------|---------|-------------------|
| Контракт | Описать в OpenAPI сценарии: embed-only register, redirect login, `returnUrl` + `state` | Страницы-обработчики, документация для клуба |
| Безопасность | CORS `CORS_ORIGIN`, валидация redirect | Не хранить секреты в клиенте клуба |
| Продукт | Решение: LK остаётся в `apps/web` или перенос в общий shell с `apps/crm` — отдельное ADR | Единый домен member, единые env (`MEMBER_FRONTEND_URL`) |

*(Детальная реализация — после утверждения доменов и политики cookie.)*

### Зачем это делать

- Клубы хотят вставить «Регистрация/Вход» на свой сайт без своей backend-разработки.
- Нужен единый auth-контур (без дублирования логики на стороне клуба).
- Нужно контролировать безопасность сессий и redirect-потоков централизованно.

### Что делать по шагам

1. Зафиксировать продуктовый сценарий: `redirect-first`, `iframe` только для регистрации.
2. Добавить в CRM раздел «Интеграция»:
   - готовые ссылки `register/login`;
   - готовый iframe snippet для регистрации;
   - опционально JS-snippet кнопки входа.
3. В API ввести проверяемые параметры `returnUrl`, `state`, `exp`, подпись.
4. В LK реализовать конечные страницы редиректов и postMessage-режим для embed.
5. Выпустить короткую публичную документацию «как встроить на Tilda/Vue/WordPress».

### Критерии готовности

- Клуб может за 5 минут вставить кнопку регистрации/входа на любой сайт.
- После логина пользователь стабильно попадает в ЛК с валидной cookie-сессией.
- Нет open-redirect, CORS и cookie-политика соответствуют браузерным ограничениям.

## 3. Epic B — Поля сущностей (fields): системные, кастомные, связи

**Расширение идеи из [MULTI_TENANT_MICROSAAS_PLAN §8](./MULTI_TENANT_MICROSAAS_PLAN.md).**

### 3.1 Таксономия полей

- **Системные, неудаляемые** — часть платформы (например `email`, идентификатор member); нельзя удалить ни на уровне портала, ни глобально; можно: label, порядок, видимость (где разрешено).
- **Системные, отключаемые на портале** — предопределены платформой, но клуб может **убрать из форм и карточек** (не показывать в UI), не удаляя определение.
- **Кастомные портала** — создаёт клуб; можно архивировать/деактивировать.

### 3.2 Типы и связи

- Типы: string, number, boolean, date, select, multi-select, email, phone, file, **relation** (ссылка на другую сущность: Product, Order, Member и т.д.).
- Для **relation**: хранение `targetEntity` + `targetId` (или отдельная таблица связей), права и фильтрация по `portalId`.

### 3.3 Хранение значений

- Варианты: `JSONB` на записи сущности + индексируемые проекции для частых фильтров; либо `field_values` (EAV-lite). Выбрать один подход и задокументировать.

### 3.4 Деактивация

- Поле `isActive` / `deactivatedAt`: скрыто везде, кроме **настроек** и при необходимости миграций данных.

### 3.5 Чеклист задач по слоям

| # | DB / Prisma | Domain | Services | API | CRM UI |
|---|-------------|--------|----------|-----|--------|
| B1 | Таблицы: `entity_definitions`, `field_definitions`, привязка к `portalId`, флаги `isSystem`, `isImmutable`, `portalHidden` | Entity/value objects, валидация | CRUD определений, резолв схемы для member/product/order | Controllers + DTO под админку/настройки | Экран «Поля» — список, порядок, видимость |
| B2 | Таблица значений или JSONB на сущности | Маппинг в доменные модели | Чтение/запись с валидацией по схеме портала | PATCH member/product с payload | Динамические формы и карточки (см. Epic F) |
| B3 | Индексы для фильтруемых custom полей | — | Поиск/фильтр | Query params / POST search | Фильтры в списках |

### Зачем это делать

- Сейчас любое новое поле = миграция + обновление DTO + UI, что тормозит внедрение.
- Клубам нужны разные анкеты member и разные карточки товара.
- Без metadata-уровня невозможно быстро запускать новые типы сущностей.

### Принцип реализации

- Сначала запуск на `member` как вертикальный slice.
- После стабилизации подхода — перенос на `product`, затем `order`.
- Валидация всегда двойная: frontend для UX, backend как источник истины.

### Definition of Done

- Админ портала может создать/скрыть/архивировать поле без деплоя.
- Форма member в CRM и (при необходимости) LK строится по схеме портала.
- По выбранным полям можно фильтровать списки на приемлемой скорости.

## 4. Epic C — Статусы (statuses): системные, кастомные, цвета, фильтры

### 4.1 Таксономия (аналогично полям)

- **Системные неудаляемые** — базовые состояния платформы.
- **Системные удаляемые/отключаемые на портале** — пресеты (например черновик для заказа).
- **Кастомные портала** — «ban», «новый», «доверенный», «premium member» и т.п.

### 4.2 Свойства статуса

- `code`, локализуемый `label`, **`color`** (hex или токен темы), порядок сортировки, `isActive` (деактивация = скрыть везде кроме настроек).
- Привязка к **типу сущности** (member, product, order, …) и к `portalId` (или null для глобальных системных).

### 4.3 Использование

- Фильтры в списках, бейджи на карточках, **включить/выключить в фильтре** (отдельный флаг в definition или в `portal_ui_config`).

### 4.4 Чеклист по слоям

| # | DB / Prisma | Domain | Services | API | CRM UI |
|---|-------------|--------|----------|-----|--------|
| C1 | `status_definitions`, связь с entity+portal, флаги immutability | Enum-like validation per entity | CRUD, разрешённые переходы (опционально state machine) | REST для настроек | Настройки статусов, color picker |
| C2 | Колонка `statusId` или FK на member/order/product (миграция со string `status` где нужно) | — | Миграция данных | — | — |

### Зачем это делать

- Строковые статусы без словаря приводят к дубликатам (`new`, `New`, `новый`).
- Нужны визуальные бейджи и предсказуемые фильтры.
- Статусы — основа для воронок, SLA и аналитики.

### Definition of Done

- Для member/product/order есть единый словарь статусов с цветом и сортировкой.
- Системные статусы защищены от удаления.
- Деактивированный статус не участвует в новых операциях, но остаётся в истории.

## 5. Epic D — Настройки портала и настроек CRM

### 5.1 Разделение

- **Настройки портала** — бренд, домены, какие документы обязательны у member, модули вкл/выкл (attendance, finance), …
- **Настройки CRM (UI)** — видимость полей/статусов в карточках и фильтрах, дефолтные таблицы.

### 5.2 Конфигурация документов member

- Связь: портал → список требуемых `Document.type` или кастомных типов; обязательность на этапе регистрации vs позже.

### 5.3 Чеклист

| # | DB / Prisma | Services | API | CRM UI |
|---|-------------|----------|-----|--------|
| D1 | `portal_settings` (JSONB или key-value), версионирование | `PortalSettingsService` | `GET/PATCH /crm/portals/me/settings` или аналог | Раздел «Настройки портала» |
| D2 | `crm_ui_settings` или `portal_ui_config` per entity | Merge с defaults | GET schema для UI | Редактор карточек/фильтров |

### Зачем это делать

- Бизнес-правила клуба (обязательные документы, включенные модули, видимость блоков) не должны быть захардкожены.
- Нужен единый источник правды для CRM UI и API поведения.

### Definition of Done

- Все критичные настройки портала читаются/пишутся через API.
- CRM UI отрисовывается согласно `portal_ui_config` без ручных условных if в коде.
- Для нового портала есть валидный набор дефолтных настроек.

## 6. Epic E — Продукты: тип (товар/услуга), категории, единицы, поля, статусы, стартовые пресеты

### 6.1 Текущее состояние

- `ProductCategory`, `MeasurementUnit`, поля цены/количества — **есть**.
- **Нет** явного `productKind` (product/service/…); **нет** `portalId` у `Product` — **нужно спроектировать** (миграция + backfill).

### 6.2 Требования

- Глобальный **тип продукта** (enum): `product`, `service`, расширяемо.
- При **подключении портала** — создавать **seed** дефолтных категорий/единиц/опционально полей (см. Epic J).
- Интеграция с Epic B/C: динамические поля и статусы для product.

### 6.3 Чеклист

| # | DB / Prisma | Services | API | CRM UI |
|---|-------------|----------|-----|--------|
| E1 | `portalId` на Product (и категории — решить: глобальные vs per-portal), `kind` enum | Каталог scoped по порталу | CRUD с фильтром | Список/форма с типом |
| E2 | Seeds для нового портала | `OnboardingSeedsService` | Вызывать из `POST /platform/portals/register` или job | — |

### Зачем это делать

- Сейчас каталог не полностью tenant-safe.
- Нужен единый подход для продажи «товара» и «услуги».
- Без начальных пресетов новый клуб долго стартует вручную.

### Важное архитектурное решение

Перед реализацией зафиксировать ADR:

- категории/единицы глобальные справочники с портальными override;
- или полностью portal-scoped справочники.

Это влияет на миграции, seeds и сложность поддержки.

## 7. Epic F — UI: карточки сущностей, фильтры, видимость

- **Карточка** — какие поля (в т.ч. custom) и статусы показывать; порядок; скрытые поля (см. Epic H).
- **Таблица** — те же правила + **колонки**.
- **Фильтры** — включить поле/статус в панель фильтра или нет.

Хранение: `portal_ui_config` (JSON schema или нормализованные таблицы `ui_field_visibility`).

**Frontend:** общий **FieldRenderer** / **StatusBadge** из `@workspace/ui`; конфиг с бэкенда.

### Зачем это делать

- Сейчас изменение состава карточки/таблицы требует доработки фронта.
- Разным ролям и клубам нужны разные представления одних и тех же данных.

### Definition of Done

- Админ в настройках включает/выключает поля и фильтры без релиза.
- Карточки member/product/order и списки синхронно используют один конфиг.
- Ошибка в конфиге не ломает страницу (есть fallback на safe defaults).

## 8. Epic G — Склад: поступления, продажи, корректировки, отчётность

### 8.1 Текущее состояние

- `Product.currentQuantity` / `initialQuantity` — ручное изменение возможно, но **нет журнала движений** и **нет** периодических поступлений.

### 8.2 Целевая модель (черновик)

- **`StockMovement`** (или `InventoryLedger`): тип `receipt`, `sale`, `adjustment`, `correction`; `quantity`, `productId`, `portalId`, `occurredAt`, `createdBy`, ссылка на `orderId`/`orderItemId` при продаже.
- **Поступление** — отдельная операция (партия), с датой и комментарием.
- **Корректировка ошибочного поступления** — отдельный тип `correction` с ссылкой на исходное движение; **не переписывать** историю — компенсирующая запись, чтобы **агрегаты за период** не искажались задним числом (или политика «только через correction»).

### 8.3 Чеклист

| # | DB / Prisma | Domain | Services | API | CRM UI |
|---|-------------|--------|----------|-----|--------|
| G1 | Таблица движений, инвариант `currentQuantity` = sum(ledger) или пересчёт транзакционно | Invariants | Запись продажи → движение; регистрация поступления | Endpoints для ledger | Журнал движений, форма поступления |
| G2 | Отчёты за период по типам движений | — | Analytics service | Export CSV / API | Экран отчёта |

### Зачем это делать

- Ручное редактирование остатка не даёт ответа «почему число именно такое».
- Финансы и склад должны быть взаимосвязаны и аудируемы.
- Нужна корректировка ошибок без потери истории.

### Ключевой принцип

История склада **append-only**:

- нельзя «тихо переписать прошлое»;
- исправления делаются компенсационными движениями;
- отчёты пересчитываются из журнала, а не из текущего остатка.

## 9. Epic H — Товар: скрытые поля, удаление, финансы

- **Себестоимость, налог** — поля с флагом `visibility: registration_only` | `finance` | `hidden` в field definition; не показывать в обычной карточке.
- **Удаление товара** — soft delete (`deletedAt`) или `isArchived`; **запрещать** hard delete если есть движения/заказы — **или** оставлять только soft.
- **Финансы:** в `FinancialTransaction` и/или в строках позиций заказа хранить **снимок**: `productName`, `sku`, `unitPrice` на момент продажи, чтобы отчёты оставались полными после «удаления» товара.

### Чеклист

| # | DB / Prisma | Services | API | CRM |
|---|-------------|----------|-----|-----|
| H1 | `OrderItem` + snapshot поля; миграция | Заполнение при создании заказа | — | Отображение истории заказа из snapshot |
| H2 | Soft delete Product | Guard на удаление | DELETE → archive | UX «архивировать» |

### Зачем это делать

- Поля типа себестоимости должны быть доступны ограниченно и не утекать в общий UI.
- Удаление товара не должно ломать старые заказы и финансовые отчёты.

### Definition of Done

- Удалённый/архивный товар не доступен в новых продажах, но полностью виден в истории.
- Любая транзакция и заказ остаются читаемыми без обращения к текущей карточке товара.

## 10. Epic I — Посещаемость: календарь-heatmap (как GitHub contributions)

- Данные: агрегат по `PresenceSession` (или по дням) — `memberId` / клуб по `portalId`.
- API: `GET` диапазон дат → массив `{ date, count }` или интенсивность.
- CRM UI: компонент сетки 7×N недель, цвет от плотности.

### Зачем это делать

- Быстрый визуальный сигнал активности клуба и сезонности.
- Отправная точка для операционных решений (часы пик, staffing).

### Definition of Done

- Есть heatmap за период (30/90/365 дней) с корректной timezone-логикой.
- Данные можно drill-down открыть до списка посещений за день.

## 11. Epic J — Seeds и платформенная админка

- **Seeds** после стабилизации схем: системные поля/статусы/категории/единицы — `prisma/seed/*.ts` или отдельная команда.
- **Platform admin** — управление глобальными определениями, порталами, приостановка клуба; отдельное приложение или `/platform/*` (уже частично есть `POST /platform/portals/register`).

### Зачем это делать

- Без seed каждый новый портал стартует в «пустоте».
- Без platform-админки невозможно централизованно обновлять системные определения и контролировать качество данных.

## 12. Приоритизация (рекомендация)

1. **PortalId + изоляция каталога** (Product/Category/Units) — иначе multi-tenant неконсистентен.
2. **Статусы + цвета** для member (замена free-form string на FK или миграция).
3. **Настройки портала** (документы, модули).
4. **Dynamic fields** — сначала member, затем product/order.
5. **Ledger склада** и snapshot в финансах/заказах.
6. Heatmap посещаемости, embed-ссылки для клубов.

### Расклад по релизам (предложение)

- **Release 1 (Foundation):** tenant-consistency каталога, словарь статусов member, portal settings (минимум).
- **Release 2 (Flexibility):** dynamic fields для member + UI config карточек/фильтров.
- **Release 3 (Operations):** inventory ledger + snapshot в orders/finance + product archive.
- **Release 4 (Growth):** external integrations (embed/links), heatmap, platform admin improvements.

## 14. Формат детальной задачи (шаблон для команды)

Для каждой конкретной задачи использовать карточку:

1. **Проблема:** что сейчас болит.
2. **Цель:** бизнес-результат и ожидаемое поведение.
3. **Scope:** что входит / что не входит.
4. **Data model:** изменения `schema.prisma`, миграции, индексы.
5. **API contract:** endpoints, DTO, ошибки.
6. **UI/UX:** экраны, состояния загрузки/ошибок, permissions.
7. **Security/Tenant:** как проверяется `portalId`, роли, guards.
8. **Acceptance criteria:** проверяемые пункты готовности.
9. **Тест-план:** unit + integration + e2e.
10. **Rollout:** фича-флаг, миграция данных, обратная совместимость.

## 15. Ключевые нефункциональные требования (для всех эпиков)

- **Tenant safety:** ни один запрос не должен возвращать данные другого портала.
- **Auditability:** изменения статусов/остатков/настроек должны быть объяснимы.
- **Backward compatibility:** миграции не ломают текущие маршруты CRM/LK.
- **Performance:** фильтры и списки с dynamic fields не деградируют критично.
- **DX:** новая конфигурация делается через админ-API, не через ручной SQL.
